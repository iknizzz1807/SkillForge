🚨 PHẦN 1: LỖI BẢO MẬT & RỦI RO NGHIÊM TRỌNG (CẦN SỬA NGAY)
1. Lỗ hổng WebSocket (Authentication Bypass)
📍 File: backend/internal/handlers/websocket_task.go & websocket_chat.go
🔥 Vấn đề: Endpoint GET /ws/task/:projectID/:userID lấy userID trực tiếp từ URL mà KHÔNG qua middleware xác thực JWT.
🛠️ Cách sửa: Lấy JWT token truyền lên qua query parameter (?token=...) hoặc sub-protocol. Trong hàm HandleConnection, viết logic parse JWT để lấy userID thực sự, từ chối nếu token sai.
2. Lưu trữ Cookie dễ bị tấn công XSS
📍 File: frontend/src/routes/login/+page.server.ts
🔥 Vấn đề: Bạn đang set cookie JWT với httpOnly: false. Nghĩa là mã độc JavaScript (XSS) trên trình duyệt có thể đọc được token này (document.cookie).
🛠️ Cách sửa: Đổi thành httpOnly: true.
3. Upload File chưa an toàn (Path Traversal & MIME type)
📍 File: backend/internal/services/files_service.go
🔥 Vấn đề: Hàm check extension avatar (.jpg, .png) chỉ check chuỗi string (tên file). Hacker có thể đổi tên file shell/php thành shell.php.jpg để vượt qua. Hàm check bảo mật strings.Contains(filename, "..") là chưa đủ.
🛠️ Cách sửa: Dùng thư viện http.DetectContentType(buffer) để đọc header thực sự của file (Magic bytes).
4. Logout hoàn toàn vô dụng ở Backend
📍 File: frontend/src/routes/api/logout/+server.ts
🔥 Vấn đề: Khi logout, bạn chỉ xóa cookie ở trình duyệt. Token JWT vẫn còn nguyên hiệu lực trên server trong 24h.
🛠️ Cách sửa: Triển khai Token Blacklist (lưu token đã logout vào Redis cho đến khi hết hạn), hoặc rút ngắn thời gian sống của Token xuống (ví dụ 15 phút) và dùng Refresh Token.
🐳 PHẦN 2: INFRASTRUCTURE, DOCKER & DATABASE
5. Mất sạch ảnh (Avatar) khi restart server
📍 File: docker-compose.yml
🔥 Vấn đề: Backend đang lưu ảnh ở ./storage/avatars bên trong container. Container restart/rebuild là mất hết dữ liệu.
🛠️ Cách sửa: Trong service backend, mount thêm volume: - ./backend/storage:/app/storage (hoặc dùng S3 như bạn đã config trong env).
6. Race Condition tạo tài khoản (Database Missing Indexes)
📍 File: backend/internal/repositories/user_repository.go
🔥 Vấn đề: Bạn check email tồn tại bằng FindUserByEmail. Nếu 2 người dùng cùng lúc bấm submit, 2 user sẽ được tạo với cùng 1 email.
🛠️ Cách sửa: Khởi tạo Unique Index cho cột email trong collection users của MongoDB khi server startup.
7. Hardcode Environment / URLs
📍 File: frontend/src/components/taskUtils.ts (Dòng 261)
🔥 Vấn đề: URL WebSocket đang bị hardcode check if (host.includes("skillforge.ikniz.id.vn")).
🛠️ Cách sửa: Đưa URL WebSocket và API vào file .env của frontend (SvelteKit cung cấp $env/static/public để lấy biến môi trường PUBLIC_WS_URL).
🚧 PHẦN 3: TÍNH NĂNG CHƯA HOÀN THIỆN (PLACEHOLDERS)

🛠️ Việc cần làm: Hoặc xóa đi để clean code, hoặc hoàn thiện việc gọi API để mở khóa tính năng.
10. Link "Find Talent" dẫn tới trang không tồn tại (404)
📍 File: frontend/src/routes/dashboard/+page.svelte (Dòng 301)

📍 File: frontend/src/routes/profile/+page.svelte

12. Gửi Email bị Comment Out
📍 File: backend/internal/services/application_service.go
🔥 Vấn đề: Chức năng gửi email khi có sinh viên apply, hoặc báo notification, đang bị bọc trong go func() nhưng comment toàn bộ ruột. Nếu mở ra mà SMTP config sai, app có thể bị crash (panic).
13. Gamification (Badges) chưa được tự động kích hoạt
📍 File: backend/internal/services/badge_service.go
🔥 Vấn đề: Có file service tính toán để trao Badge, nhưng có một đống TODO: Triển khai hàm này. Hiện tại hệ thống không tự động thưởng Badge cho user khi họ làm xong dự án.
💡 PHẦN 4: BEST PRACTICES & CODE OPTIMIZATION
14. Lỗi N+1 Query (Hiệu suất Database cực kém)
📍 File: backend/internal/services/project_service.go (Hàm GetStudentsByProjectID)
🔥 Vấn đề: Để lấy thông tin sinh viên trong dự án, bạn fetch được list studentIDs, sau đó dùng vòng lặp for để gọi hàm FindUserByID liên tục. Nếu project có 20 sinh viên, bạn tốn 21 lần query DB.
🛠️ Cách sửa: Viết một query trong MongoDB sử dụng toán tử $in: bson.M{"_id": bson.M{"$in": studentIDs}} (Chỉ tốn đúng 1 lần query). Áp dụng tương tự cho hàm lấy danh sách Projects.
15. Không có Phân trang (Pagination)
📍 File: backend/internal/repositories/project_repository.go (Hàm FindAllProjects)
🔥 Vấn đề: Lấy TOÀN BỘ project trong DB trả về cho Marketplace. Nếu có 10,000 project, hệ thống sẽ sập bộ nhớ (OOM - Out of Memory) và Frontend sẽ đứng hình.
🛠️ Cách sửa: Thêm limit và skip (hoặc cursor-based pagination) vào API GET /api/projects?page=1&limit=20.
16. Thiếu Transactions khi thao tác với nhiều Collection
📍 File: backend/internal/services/project_service.go (Hàm AddStudentToProject)
🔥 Vấn đề: Bạn Insert vào project_student, sau đó đếm số lượng và Update vào trường current_member của projects. Giả sử bước 2 bị lỗi (mất mạng/DB rớt), dữ liệu sẽ bị lệch (Sinh viên đã vào nhóm nhưng current_member không tăng).
🛠️ Cách sửa: Sử dụng MongoDB Transactions (yêu cầu Replica Set) để đảm bảo "Thành công tất cả, hoặc thất bại thì Rollback lại từ đầu".
17. AI Model Download lúc Runtime
📍 File: ai/matching/model.py (Hàm generate_score)
🔥 Vấn đề: Khởi tạo SentenceTransformer('all-MiniLM-L6-v2') bên trong hàm request. Nếu model chưa được cache, user gọi API sẽ phải chờ tải vài trăm MB. Ngay cả khi đã cache, việc load model vào bộ nhớ cho MỖI request là cực kỳ ngốn RAM và siêu chậm.
🛠️ Cách sửa: Load Model ra thành biến Global/Singleton khi khởi động FastAPI server, lúc gọi hàm chỉ truyền text vào để infer (dự đoán).
18. Báo lỗi phía Client bằng Alert ()
📍 File: frontend/src/routes/project/[project_id]/+page.svelte (Dòng 345, 386)
🔥 Vấn đề: Bắt lỗi API trả về alert("Failed to update project: " + error);. Điều này làm giảm tính chuyên nghiệp của UX.
🛠️ Cách sửa: Sử dụng Toast Notification components (như Svelte-French-Toast) hoặc Modal error mà bạn đã thiết kế sẵn.
